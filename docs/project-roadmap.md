# Project Roadmap

Last audited: 2026-06-04

This document tracks what still needs to be done across product, public content, admin workflow, research operations, and documentation. It is meant to sit above the generated research queue and below the product brief.

## Current State

The project is a working alpha of the Hallmarks of Aging LEV Tracker.

- The public app builds and has routes for overview, hallmarks, tracks, interventions, studies, findings, activity, methods, state-of-the-field notes, and admin review.
- The homepage implements the current wireframe direction: overall outlook, hallmark grid, truncated recent changes, stage badges, and trust notes. Date-specific LEV speculation now belongs on the standalone scenario route rather than the homepage outlook model.
- The file-backed admin review workflow exists for staged updates, review comments, evidence reviews, promotion readiness, and public updates.
- The operational triage layer is active: `ops/triage-state.v1.json` is regenerated by `npm run sync:research-planning` and chooses the next agent-facing work item across editorial, research, coverage repair, and data hardening.
- The research planning layer is active: `research/state/coverage-status.v1.json` and `research/backlog/track-priority.v1.json` are regenerated by `npm run sync:research-planning`, including baseline-review, field-change, and coverage-repair queues.
- Track coverage is still early, but every hallmark group now has at least one anchor baseline and all 38 seeded tracks have baseline public outlooks. No track remains ready in the baseline-review queue. Internal coverage assessments have started to capture source-landscape completeness separately from public outlook claims.

## Priority 1: Editorial Rollup and Outlook Workflow

The public record pipeline is strong for promoted evidence records, but the higher-level editorial layer still needs a first-class workflow. Today, source, study, finding, outlook, intervention, and activity records move through staged updates, evidence reviews, promotion readiness, and public updates. Curated content under `data/content/`, including state-of-the-field editions and hallmark insight copy, validates against schemas but does not yet have the same public-update trail or triage visibility.

Goal: make public summary surfaces update deliberately after meaningful public evidence changes, without forcing every field change check to edit narrative copy.

Status on 2026-06-04: first-pass implementation is in place. `editorial_rollup` is a valid generated triage mode, rollup triggers are documented, the public outlook legend now points to the method explanation, curated content schemas require review metadata, and existing curated content has a conservative `2026-05-30` review baseline. The current generated triage item still correctly asks for a June editorial rollup because June publication activity is newer than that baseline.

Planned work:

1. Add `editorial_rollup` as a triage mode.
   - Update `schemas/work-triage.schema.json` so `editorial_rollup` is a valid `workMode`.
   - Update `scripts/seed-work-triage.mjs` to generate a rollup item when the public editorial layer may be stale.
   - Proposed priority order: `publish`, `editorial_review`, `editorial_rollup`, `bootstrap`, `coverage_repair`, `surveillance`, `coverage_assessment_backfill`, `data_normalization`, `schema_hardening`, `docs_sync`, `app_surface_check`.
   - Default action: review overall outlook, state-of-field notes, hallmark insight copy, and outlook-method surfaces before starting more routine research.

2. Define rollup triggers.
   - Latest `publication_event` with an affected outlook is newer than the latest state-of-field edition.
   - Overall LEV outlook `last_updated` is older than meaningful hallmark or track outlook changes.
   - No state-of-field edition exists for the current month after material publication activity.
   - A hallmark outlook changed but the corresponding hallmark insight copy has not been reviewed.
   - A public navigation or legend element implies an outlook surface that is not actually present.

3. Add an editorial rollup runbook.
   - Create `docs/editorial-rollup.md`.
   - Cover inputs, trigger checks, expected outputs, evidence-review requirements, and post-publish verification.
   - Clarify that rollup work should not invent new evidence claims; it reconciles public interpretation with already reviewed public records.

4. Decide how content records publish.
   - Current formal publisher supports `source`, `study`, `intervention`, `finding`, `outlook`, and `activity_item`, but not `data/content/state-of-the-field/*` or `data/content/hallmark-insights.json`.
   - Choose between extending staged updates to support content records or creating a separate reviewed content-publish path.
   - Acceptance condition: state-of-field and hallmark insight updates have an auditable review trail, not just direct file edits.

5. Add review metadata to curated content.
   - Add or otherwise track `last_reviewed`, `review_reason`, and related public-update or outlook IDs for hallmark insight copy.
   - Consider similar metadata for state-of-field editions if the content-publish path needs it.
   - Keep schemas strict enough that generated planning can determine whether rollup is needed.

6. Resolve the outlook surface gap.
   - Current state: the public methods route and shell legend were removed to reduce inward-facing navigation.
   - Decide whether to build a real outlook page/panel backed by outlook records.
   - If an outlook surface is built, define the data contract before adding UI: evidence stage, interpretation note, what would change the rating, evidence gaps, evidence boundaries, and last reviewed date.

7. Integrate with publication workflow.
   - After publishing an outlook-changing staged update, run `npm run sync:research-planning`.
   - If triage emits `editorial_rollup`, handle it before ordinary field change checks unless the user explicitly chooses another task.
   - Update `docs/publication-checklist.md` to mention the rollup check as a post-publish step once the triage mode exists.

Acceptance criteria:

- `npm run sync:research-planning` can surface editorial rollup as a ranked work item.
- `ops/triage-state.v1.json` explains why rollup is needed and which public surfaces are stale.
- Overall outlook, state-of-field, hallmark insight copy, and outlook-method UI have explicit update rules.
- Content updates are either integrated into staged updates or covered by a documented reviewed publish path.
- The public site no longer implies a richer outlook surface than it actually provides.
